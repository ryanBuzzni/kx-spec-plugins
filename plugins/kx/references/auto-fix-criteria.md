# Auto-Fix 판정 기준

각 발견에 `[auto-applicable]` 또는 `[needs-judgment]` 태그를 부여하기 위한 기준.

## 원칙

- **[auto-applicable]**: 기계적 수정 — 시니어 엔지니어가 토론 없이 즉시 적용할 수정
- **[needs-judgment]**: 설계 판단 필요 — 합리적 엔지니어 간 이견 가능성 있는 수정

기본 성향:
- `[blocker]`는 **[needs-judgment]로 기우는 게 안전** (영향 범위가 큼)
- `[issue]` / `[suggestion]`은 **[auto-applicable]이 다수**

## 분류 예시

### [auto-applicable] — 즉시 적용 가능

- 누락된 `await` / `async` 키워드 추가
- 누락된 null/undefined 가드 추가 (단일 라인)
- 빈 `catch` 블록에 로깅 추가
- `finally` / context manager 누락 보완 (리소스 해제)
- N+1 쿼리를 `WHERE IN` / `includes` / `joinedload`로 일괄 조회 치환
- 매직넘버를 명명 상수로 추출 (의미 명확한 경우)
- 사용되지 않는 변수/import 제거
- 명백한 오타 / typo 수정
- 함수 반환 타입이 명확한데 빠진 경우 보완
- 동일 파일 내 명백한 코드 중복 한 곳으로 통합 (≤ 20줄)

### [needs-judgment] — 설계 판단 필요 (사용자에게 묻지 않음)

- 보안 정책 (인증/인가 모델, allowlist 설계, XSS 이스케이프 전략)
- 데이터 무결성 영향 (마이그레이션, DELETE 조건, 트랜잭션 경계)
- 상태 머신 / 상태 전이 재설계
- public API 시그니처 변경
- 사용자 가시 동작 변경 (UI 텍스트, 에러 메시지, 응답 포맷)
- 큰 리팩토링 (>20줄 수정, 여러 파일에 걸친 변경)
- 기능 제거 / 단순화 결정
- Enum/상수 완전성 — 누락된 소비처에서 어떻게 처리할지 결정 필요
- 성능 vs 가독성 트레이드오프 (캐시 도입, 인덱스 추가 등)
- 동시성 모델 변경 (락 도입, 트랜잭션 격리 수준)

## 회색 지대 처리

다음은 [needs-judgment]로 분류:

- 수정 범위가 20줄을 초과
- 동일 변경이 3개 이상 파일에 걸침
- 외부 인터페이스(API/스키마/이벤트)에 영향
- 테스트 동작이 바뀔 가능성

## 개선 사이클에서의 활용

- `[auto-applicable]` 발견은 개발 에이전트가 한 번에 일괄 패치 후 재리뷰
- `[needs-judgment]` 발견은 **사용자에게 묻지 않는다.** 대신 해당 기능의 spec 문서(`specs/[feature-name]/spec.md`)에 `## 리뷰 이슈 참고` 섹션을 만들어(이미 있으면 누적) 다음 형식으로 기록한다:

  ```markdown
  ## 리뷰 이슈 참고

  ### [YYYY-MM-DD] <한 줄 요약>
  - **유형**: needs-judgment
  - **위치**: <파일경로:라인 또는 심볼명>
  - **현상**: <무엇이 애매한지 / 어떤 판단이 필요한지>
  - **선택지**:
    - A안: <설명>
    - B안: <설명>
  - **추천**: <A안/B안 + 근거>
  - **현재 적용**: <리뷰 시점에 어떤 상태로 두었는지 — 변경 안 함 / A안 임시 적용 등>
  ```

  spec 폴더를 특정할 수 없으면(단독 리뷰 등) 현재 작업 디렉토리에 `REVIEW_NOTES.md`를 생성하여 동일 형식으로 기록한다.
- 결과: 자가평가 사이클은 `[auto-applicable]`만으로 진행되며, 사용자 인터럽트 0. 사용자는 리뷰 종료 후 `리뷰 이슈 참고` 섹션을 검수하여 후속 판단을 내린다.
