# 디자인 컨셉 일관성 확보 (Design Discovery)

프론트엔드(web/app) 에이전트는 EXTRACT 직후, CODE 시작 전에 이 절차를 수행한다.

## Step 1: 유사 기능/컴포넌트 탐색
- 대상: 모달/바텀시트, 검색, 인풋, 셀렉트, 버튼, 리스트, 카드, 폼, 토스트, 드롭다운, 빈 상태, 페이지네이션 등
- 탐색: Serena `find_symbol` / `search_for_pattern` → 공통 컴포넌트 디렉토리 (`components/common`, `ui`, `shared`, `design-system`) → 유사 화면의 마크업·스타일 패턴
- 확인 항목: 색상·간격·폰트·라운드·그림자·애니메이션·인터랙션(hover/press/disabled)·반응형/플랫폼 분기·다크모드

## Step 2: 재사용 판단

| 상황 | 처리 |
|------|------|
| 동일/유사 재사용 컴포넌트 존재 | 재사용 (신규 생성 금지) |
| 유사 패턴 1곳에만 존재 | 기존 패턴 그대로 모방 |
| 유사 패턴 **2곳 이상**, 재사용 컴포넌트 없음 | **공통 컴포넌트 추출 후 작업** (Step 3) |

## Step 3: 공통 컴포넌트 추출 시 회귀 방지 절차
1. **베이스라인 유닛 테스트 작성** — 기존 사용처의 렌더·prop 분기·이벤트·접근성 속성 (RN은 iOS/Android mocking 포함)
2. **베이스라인 통과 확인**
3. **추출/리팩토링** — 공통 컴포넌트 분리 + 기존 사용처 교체
4. **회귀 테스트 실행** — 베이스라인 재실행, 실패 시 `_workflow.md`의 Self-Eval Loop 적용
5. **신규 기능 구현** — 회귀 통과 후에만 진행
6. **신규 사용처 테스트 추가**

## Step 4: 보고에 포함
- 참조한 기존 컴포넌트/패턴 (파일 경로)
- 재사용 / 신규 작성 / 공통 추출 중 어떤 것인지
- 공통 추출 시: 신규 컴포넌트 경로, 마이그레이션된 사용처 목록, 회귀 테스트 결과
