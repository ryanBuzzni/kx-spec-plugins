# 스펙 문서 작성 스킬

현재 세션에서 조사하고 논의한 내용을 기반으로 스펙 문서를 작성한다.

> **🔵 필수 참조: 실행 전에 반드시 읽고 전체 워크플로우 흐름을 숙지한다.**
> - `~/.claude/skills/spec/references/workflow.md` — 전체 워크플로우 흐름, 자가 평가 사이클, 완료 조건

## 0. 플래그 확인

- `--skip-confirm` 플래그가 전달된 경우: **3단계(사용자 확인)를 건너뛰고** 즉시 스펙 문서를 생성한다.
- 플래그가 없는 경우: 기존대로 사용자 확인 후 진행한다.

## 1. 기능명 및 작업 영역 확인

- 사용자 메시지에서 기능명을 추출한다. 불명확하면 물어본다.
- 폴더명 형식: `YYYYMMDD-기능명` (오늘 날짜 사용)

## 2. 세션 컨텍스트 수집

현재 대화에서 다음 정보를 추출한다:

- **논의된 기능/작업 내용**: 사용자가 요청하거나 논의한 기능
- **조사 결과**: 코드 탐색, 문서 조회 등으로 파악한 정보
- **기술적 결정사항**: 선택된 기술, 라이브러리, 패턴
- **제약사항**: 발견된 제한 사항이나 주의점
- **변경된 파일**: git diff나 작업 중 수정된 파일 (있다면)

## 3. 사용자 확인

추출한 내용을 요약하여 보여준다:

```
## 세션에서 수집한 내용

**기능**: [기능명]
**작업 영역**: [백엔드/프론트엔드/풀스택]
**논의 내용**: [요약]
**기술 결정**: [결정사항]
**제약사항**: [제약사항]

이 내용으로 스펙 문서를 작성할까요?
빠진 내용이나 추가할 내용이 있으면 말씀해주세요.
```

→ **사용자 확인 후 진행**

## 4. 스펙 문서 생성

### 4.1 폴더 생성

`specs/[폴더명]/` 폴더를 생성한다.

### 4.2 spec.md 작성

```markdown
---
name: [기능명]
related_features: [관련 feature 목록]
status: planned | in_progress | completed
created: YYYY-MM-DD
---

# [기능명] Specification

## User Story
- As a [사용자 유형]
- I want [원하는 기능]
- So that [얻고자 하는 가치]

## Acceptance Criteria
- [ ] [세션에서 논의된 기준 1]
- [ ] [세션에서 논의된 기준 2]
- [ ] [세션에서 논의된 기준 3]

## Technical Constraints
- [사용 기술 스택]
- [제약사항]

## Data Structure / API
- [논의된 데이터 구조]
- [API 엔드포인트]

## Implementation Notes
- [세션에서 발견한 특이사항]
- [주의점]

## Test Plan

### 테스트 영역
- **영역**: [back/web/app — 해당하는 것 모두 나열]

### 테스트 시나리오
작업 내용에 맞는 구체적인 테스트 시나리오를 작성한다.

#### 핵심 시나리오 (반드시 통과)
- [ ] [시나리오 1: 사용자 행동 → 예상 결과]
- [ ] [시나리오 2: 사용자 행동 → 예상 결과]

#### 엣지 케이스
- [ ] [빈 입력 / 경계값 / 에러 상황 → 예상 결과]

#### Auth 처리
- **백엔드**: 재사용가능한 access token 생성 헬퍼 메소드
- **웹 E2E**: storageState로 1회 로그인 후 재사용
- **앱**: Maestro 공유 로그인 서브플로우
- **Auth 불가 시**: `test/` 페이지 별도 생성

### 실행 방법
- `testing [영역]` — 자동 테스트
- `testing [영역] [시나리오]` — 시나리오 기반 테스트
```

### 4.3 plan.md 작성

로드한 워크플로우 레퍼런스의 Phase-Based Implementation을 따라 구현 계획을 수립한다.

```markdown
# [기능명] Implementation Plan

## Overview
- **작업 영역**: [백엔드/프론트엔드/풀스택]
- **워크플로우**: [적용된 워크플로우 레퍼런스]
- **예상 Phase 수**: [N]

## Phase 1: [단계명]
- **목표**: [이 단계에서 달성할 것]
- **작업 항목**:
  - [ ] [구체적 작업 1]
  - [ ] [구체적 작업 2]
- **검증**: [확인 방법]
- **상태**: [ ] Not Started

## Phase 2: [단계명]
- **목표**: [이 단계에서 달성할 것]
- **작업 항목**:
  - [ ] [구체적 작업 1]
  - [ ] [구체적 작업 2]
- **검증**: [확인 방법]
- **상태**: [ ] Not Started

## Phase 3: [단계명]
...

## Phase N: 테스트
- **목표**: 구현된 기능에 대한 테스트 코드 작성 및 검증
- **테스트 영역**: [back/web/app — 작업 영역에 따라]
- **테스트 시나리오**:
  - [ ] [시나리오 1: 사용자 행동 기반 설명] — 예상 결과
  - [ ] [시나리오 2: 사용자 행동 기반 설명] — 예상 결과
  - [ ] [시나리오 3: 엣지 케이스/에러 케이스] — 예상 결과
- **Auth 처리**: [필요 시 방법 명시]
- **실행 방법**: `testing [영역]` 또는 `testing [영역] [시나리오]`
- **상태**: [ ] Not Started

## Notes
- 구조적 변경과 동작 변경은 별도 커밋
- 각 Phase 완료 후 context.md 업데이트
```

### 4.4 context.md 작성

```markdown
# [기능명] Context

## 마지막 업데이트
- 날짜: YYYY-MM-DD
- 세션 상태: Planning

## 현재 진행 상태
- Phase: Phase 1 - [단계명]
- 작업: plan.md 작성 완료, 구현 대기
- 상태: 완료 대기

## 완료된 작업
- [x] spec.md 작성
- [x] plan.md 작성
- [x] context.md 작성

## 다음 작업 (새 세션에서 바로 시작)
- [ ] Phase 1 첫 번째 작업 항목

## 중요 결정 사항
- [세션에서 논의된 결정사항]

## 변경된 파일
- `specs/[폴더명]/spec.md` - 요구사항 정의
- `specs/[폴더명]/plan.md` - 구현 계획
- `specs/[폴더명]/context.md` - 상태 추적

## 알려진 이슈
- [있다면 기록]
```

## 5. 결과 출력

```
스펙 문서 생성 완료:
  - specs/[폴더명]/spec.md (요구사항)
  - specs/[폴더명]/plan.md (구현 계획)
  - specs/[폴더명]/context.md (상태 추적)

적용된 워크플로우: [백엔드/프론트엔드/풀스택]
```

## 6. 다음 단계 자동 연결

- **spec-plan에서 호출된 경우** (`--skip-confirm` 플래그): 스펙 문서 생성 완료 후 **즉시 `/kx:spec-execute` 스킬을 실행**한다. 멈추지 않는다.
- **단독 실행인 경우**: 다음 안내를 출력하고 종료한다.
  ```
  다음 단계:
    - /kx:spec-execute → 개발 실행
    - testing [영역] → 테스트 실행
  ```
