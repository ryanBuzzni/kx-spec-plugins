# 인터뷰 흐름 가이드

spec-plan의 **3. 질문 단계** 세부 절차. 한 번에 한 질문, 비개발자 친화 언어, AI 추측 옵션 우선.

---

## 핵심 원칙

- **One question at a time** — 한 메시지에 단 하나의 질문만
- **AskUserQuestion 우선** — AI 추측 옵션 2~4개 + Other. 자유 텍스트는 폴백
- **비개발자 언어** — "화면/버튼/동작/결과". "API/컴포넌트/라이브러리" 같은 용어 금지
- **의도 위주, 구현 위주 X** — "왜/뭐가 일어나야 하나"만. "어떻게 만들지"는 묻지 않음
- **이미 명확한 단계는 스킵** — 추측 가능하면 건너뜀

---

## 단계별 흐름

### [3-0] 목표 명확성 체크

사용자 메시지에 동사+대상이 없거나 "~하고 싶어/~필요해" 수준일 때만 트리거.

- AI가 추측한 목표 2~3개를 옵션으로 제시 + Other
- 명확하면 스킵하고 [3-1]로

예) "즐겨찾기 좀 손보고 싶어"
→ "어떤 게 가장 가까워요?" (옵션: 새로 추가 / 동작 변경 / 버그 수정 / Other)

### [3-1] 작업 유형 분류

자동 판별:
- **신규**: 없는 기능을 새로 만듦
- **수정**: 기존 기능 동작을 바꿈
- **버그**: 의도와 다르게 동작하는 걸 고침
- **리팩토링**: 사용자 체감 동작은 같고 코드만 정리

모호하면 단일 객관식 질문. 명확하면 스킵.

### [3-2] 대상·범위

사용자 관점 스코프만 묻는다. "어디서 / 누구한테 / 언제".

질문 예시는 `question-banks.md` 참고.

- 신규: 가볍게 (한두 개)
- 수정: 중심 단계 (호환성/범위 중요)
- 버그: 재현 환경 위주
- 리팩토링: 영향 범위 위주

### [3-3] 시나리오

"무엇이 일어나야 하나"를 골든 패스 → 엣지/에러 순으로.

- 골든 패스: "사용자가 X 하면 Y가 일어나야 함" 형태
- 엣지: AI가 추측한 동작 시나리오를 옵션으로 제시
  - 예) "비어 있을 때는 어떻게 보여요?" — (옵션: 안내 문구 / 빈 영역 / 자동 숨김 / Other)
- 각 시나리오는 별도 질문으로

작업 유형별:
- 신규: 무겁게 다룸
- 수정: 변경되는 동작만
- 버그: 재현 시나리오 = 이 단계의 전부
- 리팩토링: 거의 스킵

**사용자 체감이 달라지는 기술 선택**(예: "저장 즉시 반영 vs 새로고침 필요")은 제품 관점 질문으로 변환해서 여기에 포함.

### [3-4] 기술·구현 — 사용자에게 묻지 않음

AI가 코드 검증 결과 + 기존 패턴 기반으로 결정한다.

- 비개발자 모드: 마지막 계획에서 "참고용"으로만 요약 (이해 가능한 언어로)
- 개발자 모드: 필요하면 객관식으로 한 번 확인 가능

---

## 개발자/비개발자 감지

사용자 메시지에 다음이 있으면 **개발자 트랙** 허용:
- 코드 스니펫, 파일 경로, 기술 용어(API, hook, store, schema 등)

없으면 **비개발자 트랙** 기본. 애매하면 단일 객관식으로 한 번 확인:

> "기술적인 부분도 같이 의논할까요, 동작 위주로 갈까요?"

비개발자 트랙에서는 [3-4]를 절대 사용자에게 묻지 않는다.

---

## AskUserQuestion 작성 규칙

- **label**: 1~5단어, 사용자 언어
- **description**: 한 줄로 의미 설명. 무엇이 달라지는지
- **옵션 수**: 2~4개. 추천 옵션을 맨 위에 두고 "(추천)" 표기
- **header**: 12자 이내 카테고리 라벨

자유 텍스트가 필요해 보여도 일단 추측 + Other로 제시. 추측이 불가능할 때만 plain text.
