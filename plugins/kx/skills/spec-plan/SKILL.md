# Spec Plan 스킬

사용자의 작업 요청을 분석하고, 불명확한 부분을 **단계별 인터뷰**로 확인한 후 구현 계획을 수립한다.

> **🔵 필수 참조: 실행 전에 반드시 읽고 전체 워크플로우 흐름을 숙지한다.**
> - `~/.claude/skills/spec/references/workflow.md` — 전체 워크플로우 흐름, 자가 평가 사이클, 완료 조건
> - `./references/interview-flow.md` — 단계별 인터뷰 절차 (3단계 진입 시 필수 로드)
> - `./references/question-banks.md` — 작업 유형별 질문 후보 (3단계 진입 시 필수 로드)

---

## 동작 방식

### 1. 요청 분석

사용자 입력을 분석하여 다음을 파악한다:
- 무엇을 만들거나 수정하려는지
- 어떤 기술/파일/모듈이 관련되는지
- 명확한 부분과 불명확한 부분 구분

### 2. 코드 기반 검증

요청과 관련된 코드를 탐색하여 분석 내용을 검증한다:
- 언급된 파일/모듈이 실제로 존재하는지
- 현재 코드 구조와 요청의 정합성
- 수정 대상의 실제 구현 상태

검증 결과를 사용자에게 먼저 공유한다:

```
## 코드 검증 결과

**확인된 현재 상태:**
- [파일/모듈 존재 여부, 현재 구조 등]

**요청과의 차이점:**
- [예상과 다른 부분이 있다면]
```

이 검증 결과는 이후 인터뷰 옵션 추측의 근거가 된다.

### 3. 단계별 인터뷰 (불명확한 부분이 있을 때)

> 진입 시 `references/interview-flow.md`와 `references/question-banks.md`를 먼저 읽고 그 절차를 따른다.

핵심 규칙:
- **한 번에 한 질문만** (One question at a time)
- **AskUserQuestion 우선** — AI가 코드 기반으로 추측한 옵션 2~4개 + Other
- **비개발자 언어** — 화면/버튼/동작/결과. 기술 용어 금지
- **이미 명확한 단계는 스킵**

인터뷰 순서:
```
[3-0] 목표 명확성 (불명확하면 단일 질문)
[3-1] 작업 유형 분류 (신규/수정/버그/리팩토링)
[3-2] 대상·범위 — 사용자 관점 스코프
[3-3] 시나리오 — 골든 패스 → 엣지/에러
[3-4] 기술·구현 — 사용자에게 묻지 않음, AI가 결정
```

사용자 체감이 달라지는 기술 선택은 제품 관점으로 변환해 [3-3]에 포함한다.
개발자/비개발자 감지 및 트랙 분기는 `interview-flow.md` 참고.

### 4. 분기

- **질문이 있을 경우** → 인터뷰 진행 후 답변 대기
- **질문이 없을 경우** → 바로 `/kx:spec-write` 스킬을 실행 (`--skip-confirm` 플래그 전달)

### 5. 답변 수신 후

답변을 반영하여 구현 계획을 제시한다:

```
## 구현 계획

### 목표
[한 줄 요약]

### 작업 목록
1. [작업 1] - [대상 파일/모듈]
2. [작업 2] - [대상 파일/모듈]
3. [작업 3] - [대상 파일/모듈]

### 기술적 접근
- [선택된 방식과 이유] — 비개발자 모드에서는 이해 가능한 언어로 요약

### 예상 영향
- [수정되는 파일/모듈]
- [주의할 점]
```

계획 승인 후 `/kx:spec-write` 스킬을 실행한다 (`--skip-confirm` 플래그 전달).
`/kx:spec-write` 완료 후 자동으로 `/kx:spec-execute` 스킬이 이어서 실행된다.

> **흐름: /kx:spec-plan → /kx:spec-write(승인 스킵) → /kx:spec-execute**

---

## 핵심 원칙

- **추측하지 않는다** — 모르면 물어본다
- **한 번에 한 질문** — 동시에 여러 질문 금지
- **객관식 우선** — AI 추측 옵션으로 답변 부담 최소화
- **비개발자도 답할 수 있는 질문** — 제품/사용자 언어만 사용
- **이미 명확하면 스킵** — 불필요한 질문 금지
