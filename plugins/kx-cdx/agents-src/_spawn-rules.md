# 에이전트 Spawn 규칙

서브에이전트를 호출(spawn)하는 모든 skill·에이전트는 이 규칙을 따른다. (플러그인 환경에서 spawn 실패·결과 누락 사례를 막기 위함.)

## 1. 호출 파라미터 명확화

서브에이전트 호출 시 다음을 명시한다.

| 파라미터 | 규칙 |
|---------|------|
| `subagent_type` | **반드시 명시** — 생략 시 기본값(`general-purpose`)이 적용되지만 의도된 에이전트가 아닐 위험. 커스텀 에이전트가 없으면 `general-purpose`를 명시 |
| `description` | 3~5단어 짧은 설명 (예: `"Backend test for department PR"`, `"Run web tester on auth flow"`) |
| `prompt` | self-contained — 부모 대화 컨텍스트 없이도 작업 가능하도록 작성 |
| `run_in_background` | **사용 금지** (foreground) — 결과를 즉시 받아 검증해야 하므로 백그라운드 실행 지양. 명시적 장기 작업이 아닌 한 false |

## 2. 검증 가능한 자기 진단

spawn 직후 결과를 **반드시** 점검한다. 침묵·낙관적 가정 금지.

- spawn 결과가 **비어있거나 `[Tool result missing]`이면 즉시 사용자에게 보고**
- "spawn 됐다"고 가정하지 말 것 — 결과 본문이 있어야 성공
- 결과가 비정상이면:
  1. 한 번 재시도(`description`/`prompt`를 명확히 다시 작성해서)
  2. 재시도도 실패면 **메인 세션이 직접 처리로 전환** — 이때 사용자에게 명시적으로 알리고 진행
- "에이전트 실행했고 잘 됐을 것" 같은 추정 보고 금지. 실제 결과 본문(작성 파일·테스트 통과 여부 등)을 인용해야 완료 인정

## 보고 시 포함할 항목
- 호출한 `subagent_type`
- 결과 길이/요약 (또는 실패 원인)
- 폴백(직접 처리)으로 전환했다면 그 사유
